This stylesheet embeds the DITA vocabulary into the XSLStyle scaffolding used to document the constructs in an XSLT stylesheet. For ease of use, any namespace that may be attached to the documentation constructs found inside of the XSLStyle constructs is stripped before the rendering stylesheets are invoked, thus keeping the XSLT author from having to worry about default namespace declarations and such simply for the documentation.

XSLStyle™ implements a methodology for styling stylesheets using a documentation vocabulary into formatted documentation and rigourous completeness reports. 1.1. How to use XSLStyle™

See the "XSLT Resources" section linked from the top of the http://www.CraneSoftwrights.com "Free resources" web page for up-to-date information on invocation.

1.1.1. Basic mechanisms

The W3C stylesheet association specification http://www.w3.org/1999/06/REC-xml-stylesheet-19990629 provides for indicating a stylesheet with which to render an XML document. This indication is done by way of an XML processing instruction.

XSLT stylesheets are XML documents so the stylesheet association processing instruction can be added to XSLT documents, suggesting a stylesheet with which to render the stylesheet.

The XSLTStyle™ stylesheet and stylesheet library provides an environment in which any documentation vocabulary can be used to annotate and XSLT stylesheet with documentation. Furthermore, the stylesheet can enforce "business rules" regarding the stylesheets, thus being a tool in development for measuring the compliance or completeness of an XSLT stylesheet according to those rules.

XSLT allows non-XSLT-namespace constructs at the top-level of an XSLT stylesheet (except in advance of <xsl:import> instructions). The embedded documentation, using any vocabulary, is wrapped by top-level elements in the XSLT stylesheet using the XSLStyle™ vocabulary. The XSLStyle™ vocabulary is detected by these stylesheets and the wrapped content is fed to the rendering stylesheets, which then render the result as HTML to the output (typically a web browser screen).

1.1.1.1. For documentation

XSLStyle™ does not, itself, define a documentation vocabulary. It is merely a structural vocabulary for decorating an XSLT stylesheet, within which a documentation vocabulary is used for the actual prose of the documentation.

These example stylesheets themselves are documented using DITA, though there is no obligation with which to do so. The XSLStyle™ stylesheet fragments (including that for DITA) are documented in DocBook The author has also utilized the Intelligence Community Metadata Standard for Publications - ICMSP vocabulary in a similar embedding mechanism with which a set of ICMSP stylesheets are documented (in that they document themselves).

Thus, the mechanism is independent of the documentation vocabulary, and all documentation vocabulary handling is localized at the top stylesheet of a stylesheet import tree, where the top documentation-specific stylesheet imports the XSLStyle™ stylesheet. This top stylesheet implements interpetation of the embedded vocabulary, either directly within the fragment or by importing the stylesheets required.

1.1.1.2. For completeness

The power of XSLT allows arbitrary business rules to be coded, using XPath addresses to find content in an XML document, and stylesheet logic to establish if the content meets the constraints of the rules.

A stylesheet writing team can choose the business rules for writing stylesheets, encode the detection and confirmation of meeting these rules in XSLT, and list any detectable incompleteness as a section of the final report.

A stylesheet writer's manager, then, could choose not to accept a stylesheet as being completely finished until the embedded documentation report is free of all business rule inconsistencies.

For a quick demonstration of this, remove any XSLStyle™ documentation construct from the stylesheet producing this report and refresh the report to see the inconsistencies section. You can remove any top-level element's documentation construct, or a template rule parameter's documentation construct in order to trigger the display of this business rule violation section. Restoring the documentation will restore the report not to have any inconsistencies section.

The inconsistency report can be suppressed by setting the invocation parameter suppress-inconsistencies to true.

1.1.2. Using the vocabulary

1.1.2.1. Stylesheet fragment stylesheet engagement

Each stylesheet with XSLStyle™ documentation, or at the least those that are at the top of an XSLT import tree, needs a processing instruction to automatically engage the stylesheet fragment when processed with XSLT. For example, the invocation of the DITA shell around XSLStyle™ at the top of this stylesheet reads as: 

<?xml-stylesheet type="text/xsl" href="xslstyle-dita.xsl"?>

Note this processing instruction is not mandatory as the stylesheet could of course be invoked explicitly in a command line shell that knows which XSLStyle™ stylesheet shell to use, but by embedding the processing instruction one could invoke the XSLT processor in the command line shell to just use the xml-stylesheet processing instruction.

1.1.2.2. Stylesheet fragment documentation

Any stylesheet with XSLStyle™ documentation must have an <xs:doc> element with an info= attribute showing metadata information, a filename= attribute having some unique filename information identifying the file (so as to be unambiguous with any other stylesheet), and a vocabulary= attribute indicating the vocabulary being embedded. These attributes must be attached to the first <xs:doc> in the document. This first <xs:doc> must also have an <xs:title> child as the overall title of the documentation.

You can choose to use any level of detail in either the info= or filename= attributes as there are no restrictions to the content. Use whatever you will find helpful in the final report to identify the fragment and understand the metadata. These stylesheets happen to use CVS source code control metadata in the information attribute. The following is an example from an earlier edition of this stylesheet: 

<xs:doc info="$Id: xslstyle.xsl,v 1.17 2005/10/17 04:58:43 G. Ken Holman Exp $"
        filename="xslstyle.xsl" global-ns="xs" vocabulary="DITA">
  <xs:title>XSLStyle™ illustration for the DITA vocabulary</xs:title>
  <p>
    XSLStyle&#x2122; implements a methodology for styling stylesheets
    using a documentation vocabulary into formatted documentation and
    rigourous completeness reports.
    Note that the use of "&#x2122;" is only to protect using this name for
    this set of stylesheets.  The use of these stylesheets is subject to
    the following license:
  </p>
  <codeblock>
Copyright (C) - Crane Softwrights Ltd.
...
</xs:doc>

There are a number of other top-level <xs:doc> attributes corresponding to external and internal stylesheets, and suppressing or expanding portions of the final report. Please see the corresponding shell stylesheet that wraps the core stylesheet supporting this vocabulary.

All other <xs:doc> elements in the stylesheet must not have these attributes, but may be present in order to indicate major sections of the stylesheet that group the top level elements for some reason, typically promoting good stylesheet maintenance. These have an <xs:title> element child used for indexing. For example, again from this stylesheet, the documentation for the next section of this report: 
<xs:doc>
  <xs:title>Global properties</xs:title>
  <p>
    These are the invocation and/or importation properties that can be set
    when either invoking the stylesheet from the command line, or using an
    importing stylesheet to import this while overriding these properties.
  </p>
</xs:doc>

If you look inside this example stylesheet you will note that sections are embedded in the documentation using <topic> elements. This environment will supply defaulted id= and class= attributes where required, so the embedded documentation need not have them unless you feel you need them. 

  <topic>
    <title>Using the vocabulary</title>
    <topic>
      <title>Stylesheet fragment stylesheet engagement</title>
      <p>
        Each stylesheet with XSLStyle™ documentation, or at the least
        ...

Note that the global-ns= and internal-ns= attributes support a business rule created for these particular stylesheets: the attributes enumerate in a space-separated list all of the namespace prefixes allowed for names of top-level constructs and modes for template matching rules. Internal variables are summarized separately from global variables. An inconsistency is reported for any top-level element violating this rule, unless the element uses ignore-ns="yes" to explicitly state that it does not belong in the namespace of the stylesheet's global constructs, as this stylesheet does for command-line invocation parameters: 

<xs:param ignore-ns='yes'>
  <p>
    Invocation/importation parameter set to true to expose the entire import
    tree instead of just the portion of the import tree that is documented
    with XSLStyle&#x2122; constructs (stopping at the point where there are
    no such constructs).
  </p>
</xs:param>
<xsl:param name="force-full-import-tree" select="false()"/>

The inconsistency report for global namespace naming violations can be suppressed by setting the invocation parameter suppress-non-ns-global-reports to true.

The content of the XSLStyle™ construct is any content that is recognized by the documentation vocabulary stylesheets rendering the report. This is the content gleaned from the stylesheet to make up the final report.

1.1.2.3. Stylsheet top-level element documentation

These stylesheets implement the business rule that each and every top-level XSLT element must be documented using an XSLStyle™ construct. This construct must have the same local name as the top-level XSLT element. It must be the immediately-preceding element (see the <xs:param> example in the previous section), or it can reside anywhere in the top-level of the stylesheet if both the XSLT element and the XSLStyle™ element have the same name= attribute value.

Furthermore, every <xs:template> element documenting an <xst:template> element must have an <xs:param> child element, with a name= attribute, for each of the template's invocation parameters. The following is an example: 

<xs:template>
  <p>
    This adds a reference to the in-memory reference tree, maintaining the
    path of the module referenced.
  </p>
  <xs:param name="path">
    <p>This is the parent's path so far, on which to build this path.</p>
  </xs:param>
  <xs:param name="href">
    <p>
      This is the <codeph>href=</codeph> attribute used by the parent to
      call this module.
    </p>
  </xs:param>
</xs:template>
<xsl:template match="*" mode="xs:collect-module-hrefs">
  <xsl:param name="path"/>
  <xsl:param name="href"/>
...

Top-level elements without names can be suppressed from the report by setting the invocation parameter suppress-unamed-construct-documentation to true.

1.1.3. Using the import/include tree

After the table of contents, the import/include tree is shown for the set of stylesheets imported and included from the stylesheet being documented. The report will stop a particular branch of the import tree when it encounters a fragment without XSLStyle™ embedded documentation, or whose embedded documentation is using a different vocabulary. A complete report of the full import tree can be engaged by setting the invocation parameter force-full-import-tree to true.

Each entry in the import tree is hyperlinked to the report's documentation for that stylesheet fragment, thus, it is only necessary to browse the "top" fragment in the import tree to see all of the documentation for all of the fragments in the entire import tree.

1.1.4. Using the index of named XSLT top-level elements

At the bottom of the report, before any inconsistencies, is a generated index of all of the named XSLT top-level elements in the entire import tree. This is alphabetized by the local name of the construct, and sub-indexed by the first letter of the local name.

Each entry is hyperlinked three ways: to the construct's entry in the report, to the construct's group heading in the report, and to the construct's stylesheet fragment documentation in the report.
